### 1. **Claude Code 新指令 /code-review --comment：AI 審完程式碼直接推 GitHub PR 留言**

**帳號：** @AnthropicAI

**類型：** 🚀 新工具

**核心方法/技巧：**
- Claude Code v2.1.147 新增 `/code-review` 指令（取代舊版 /simplify），在 session 中直接叫出 AI 代碼審查
- 加上 `--comment` 參數後，AI 的審查意見會自動推送到 GitHub PR 的 inline 留言，省去複製貼上的步驟
- `/usage` 指令升級：現在分類顯示 Skills / Subagents / Plugins / MCP server 各別用量，一眼看清哪個環節燒最多 token
- `Ctrl+T` 新快捷鍵：釘選背景 session，更新時自動重啟不中斷當前工作

**為什麼有用：**
對任何使用 Claude Code 協作開發或審查內容的人來說，/code-review --comment 把「AI 看完→我手動貼留言」壓縮成一個指令，PR 審查工作流大幅提速。

**編輯觀點：** 💬 這個功能讓 Claude Code 從「個人助理」升級成「真正的 PR 協作者」，往後的 code review 文化恐怕要被 AI 留言淹沒了。

**天馬行空的延伸：** 🚀 如果 /code-review --comment 能連結部落格草稿的 GitHub repo，AI 就能直接在你的文章 PR 上留「段落不通順」「這個比喻不夠精準」——成為你的自動文稿編輯機器人。

**💬 最有價值的回應：**
> "The --comment flag is the feature I didn't know I needed. Spent 20 mins copy-pasting AI review notes into GitHub last week. This week: one flag."
— @bcherny（Claude Code 核心貢獻者，補充了 --comment 的實際節省時間數據）

**推文連結：** https://github.com/anthropics/claude-code/releases/tag/v2.1.147

### 2. **guizang-ppt-skill 突破萬星：25 天讓 Claude Code 生成有設計感的 AI 簡報**

- **大多數人用錯了 Codex**：只拿來改小 diff，但真正的價值在架構層級的大改動——換框架、重構模組、跨檔批次修改

- 搭建自動化測試 harness，讓 Codex 自行驗證輸出，減少人工確認迴圈

**為什麼有用：**
RUNBOOK 模式直接解決「AI 做到一半跑偏」的問題，/goal 讓 Codex 能處理需要多輪對話才能完成的複雜任務——這是 Codex 從玩具到生產工具的關鍵配置。

**編輯觀點：** 💬 「大多數人只用 Codex 改小 diff」這句話點醒了我：我們都在用超跑開計程車。

**天馬行空的延伸：** 🚀 把 RUNBOOK 模式用在內容創作——「本季度內容計劃 RUNBOOK」交給 AI，讓它依據規格書自動產出文章大綱、素材清單、發布時程，內容策略師從此只做 RUNBOOK，不做執行。

**💬 最有價值的回應：**

### 4. **阮一峰週刊第 397 期：4 款工具讓 AI 工作流再進化（水印移除、Agent 界面、手機遠端、有聲書）**

**帳號：** @ruanyf（阮一峰）

**類型：** 🚀 新工具

**核心方法/技巧：**
- **Remove-AI-Watermarks**：命令行工具 + Python 套件，可移除 AI 圖片的可見及不可見水印（含 Google SynthID），對需要「乾淨」AI 圖片的創作者極為實用
- **AVC（Agent View Controller）**：將 AI Agent 在 terminal 的確認彈窗轉為瀏覽器互動式 pop-up，讓你跑長任務時用正常介面管理，而非盯著終端機
- **Lucarne**：讓本機 AI Agent 連接微信／Telegram，離開電腦時從手機接收進度更新、批准決策、繼續 session
- **Abogen**：跨平台桌面應用，將 ePub／PDF／txt 轉換為有聲書，支援多語言 TTS

**為什麼有用：**
這四個工具解決的都是 AI 使用者的「邊界痛點」：圖片版權、長任務監控、離線管理、內容消費方式——每一個都能立刻裝起來用。

**編輯觀點：** 💬 Lucarne 是本週最被低估的工具：能從手機管理 AI Agent 任務意味著你的 AI 工作流第一次真正不需要「坐在電腦前」。

**天馬行空的延伸：** 🚀 Lucarne + 長跑 Agent + Abogen 組合：讓 AI Agent 在你睡覺時自動研究主題、生成報告，早上用手機批准，起床就有 AI 幫你唸好的有聲摘要。

**💬 最有價值的回應：**
> "AVC 讓我第一次敢把 Agent 的確認頻率設很低——以前怕漏掉重要確認，現在有瀏覽器 pop-up 就放心多了。"

### 6. **「不要外包學習」：Addy Osmani 的 AI 時代認知投降警告**

**帳號：** @addyosmani

**類型：** 📝 提示技巧

**核心方法/技巧：**
- **認知投降（Cognitive Surrender）** 的定義：你完全停止思考，盲目接受 AI 給的答案——這才是真正的危機
- 讓 AI 修 bug 卻不理解原因 = 技術停滯：六個月後你不會比今天更強
- 正確用法：用 AI 做輔助，每次修改都要親自理解原因——問 AI「為什麼這樣改」，而非只問「怎麼改」
- 學習過程不能外包，即使在 AI 時代——尤其在 AI 時代

**為什麼有用：**
對內容創作者和 AI 使用者的精準提醒：AI 加速執行的同時，也在加速技能退化的速度。理解每一個 AI 輸出背後的邏輯，才能真正提升自己的能力。

**編輯觀點：** 💬 Addy 說的認知投降不只發生在工程師身上——讓 AI 寫完文章自己都沒讀懂的內容創作者，同樣在投降。

**天馬行空的延伸：** 🚀 每週設一個「無 AI 日」：同樣的工作不用 AI 自己做一遍。這不是懷舊，而是校準——知道沒有 AI 你能做到哪裡，才知道 AI 幫你跨越了什麼。

**💬 最有價值的回應：**
> "The difference between a junior who uses AI to learn and a junior who uses AI to avoid learning will compound dramatically over 2 years. One will be irreplaceable. One will be replaced."
— @steipete（從工程師職涯角度補充了長期影響，極具說服力）

### 7. **AI 自動化悖論：Dan Shipper 的反直覺觀察——自動化越多，需要的人反而越多**

**帳號：** @danshipper

**類型：** 💡 工作流優化

**核心方法/技巧：**
- Every（danshipper 的公司）用 AI Agent 自動化了「每一件能自動化的事」，員工卻從 4 人增長到 30 人
- 原因：自動化不縮小工作範圍，而是**擴大**了工作範圍——能做更多事，就有更多事值得做
- 對內容創作者的啟示：AI 處理生產量（量），人負責策展、品味、策略和關係（質）——這些正是新的成長驅動力
- 實用框架：「不要把任務完成和工作消失混為一談」

**為什麼有用：**
這個觀察打破了「AI 會讓你失業」的線性邏輯，給出了更符合實際的框架：AI 是能力放大器，而非替代品。理解這一點，對如何分配自己的學習投資極為重要。

**編輯觀點：** 💬 這個悖論的另一個名字叫「傑文斯悖論」——效率提升往往增加總需求，而非減少它。蒸汽機不是讓人少工作，而是讓更多人開始工作。

**天馬行空的延伸：** 🚀 如果 AI 自動化讓公司從 4 人擴張到 30 人，下一個問題是：新增的 26 個人在做什麼？答案可能是「以前不可能做、現在 AI 使其可行的事」——這個清單就是你的機會清單。

**💬 最有價值的回應：**
> "We've seen this at every company I've advised. Teams that automate the most, end up hiring the most. It's the Jevons Paradox for knowledge work."
— @levie（Aaron Levie，Box CEO，同週相互印證的類似觀察，背書效果強）

**推文連結：** https://x.com/danshipper（2026-05-21）

### 8. **「所有模型實驗室都是 Agent 實驗室」：swyx 的 2026 Agent 產品堆疊公式**

**帳號：** @swyx（Shawn Wang）

**類型：** 🛠️ 可複用方法

**核心方法/技巧：**
- **2026 Agent 產品完整堆疊**：Model + Harness + Workflow + UI + Memory + Economics——模型品質已不是護城河，Harness（如何運行 Agent）才是
- **新企業 AI 三角**：Harness + Eval + Deployment——沒有這三個，模型再好也無法進企業
- **時代切換定義**：「2024 年的問題是『我能不能 build 一個 Agent』，2026 年的問題是『我能不能 ship 一個 Agent 產品』」
- **MCP 2026 RC 更新**：協議改為無狀態（無 session ID、無握手），更易於擴展和負載均衡

**為什麼有用：**
給所有想用 AI 建構工具或工作流的人一個清晰的全局框架——下次評估一個 AI 工具時，對照這個堆疊清單，就能快速找到它的弱點在哪。

**編輯觀點：** 💬 「能 build vs. 能 ship」這個切換是 2026 年最重要的能力分水嶺。Harness + Eval + Deployment 的人才稀缺度，遠超 prompt 工程師。

**天馬行空的延伸：** 🚀 把這個堆疊用在個人 AI 工作流設計上：你有 Model（選哪個 AI），但你有 Harness（怎麼跑它）、Eval（怎麼驗證輸出品質）、Memory（跨 session 記住什麼）嗎？逐一補齊，就是個人 AI 能力的升級路徑。

**💬 最有價值的回應：**
> "The economics layer is what everyone skips. Most AI workflows have no cost model — they just keep calling APIs until the bill is due."
— @mattturck（VC 視角補充了 Economics 層的重要性，是整個堆疊中最容易被忽略的一環）

**推文連結：** https://www.latent.space/p/ainews-all-model-labs-are-now-agent（2026-05-23）
