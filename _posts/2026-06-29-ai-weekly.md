### 1. **Karpathy CLAUDE.md 進化：十條黃金規則，讓 AI 學會監控自己的思維**

**帳號：** @karpathy（Anthropic 預訓練團隊）

**類型：** 🛠️ 可複用方法

**核心方法/技巧：**
- 前四條已知規則：先思後寫、簡潔優先、外科手術式修改、目標驅動執行
- 新增六條自我檢查規則：Verification（先寫測試再修 Bug）、Debugging（讀完完整錯誤訊息才動手）、Dependencies（每個新套件都是永久承諾，先問標準庫能否勝任）
- 新增四大失敗模式命名：Kitchen Sink（修龍頭卻翻新廚房）、Wrong Abstraction（同邏輯出現三次才應抽象）、Optimistic Path（只寫快樂路徑）、Runaway Refactor（一個檔案變成十個）
- CLAUDE.md 注入機制：每次 Claude Code 啟動時自動注入對話 context，不影響 system prompt

**為什麼有用：**
前四條規則教 AI 怎麼寫程式，新增六條規則教 AI 怎麼監控自己的推理——在無人盯守的自動化 loop 中，這六條比前四條更重要。任何用 Claude Code 跑 `/loop` 或 `/goal` 的人都應該立刻更新 CLAUDE.md。

**編輯觀點：** 💬 「Kitchen Sink」和「Runaway Refactor」這兩個失敗模式命名太準了——給錯誤一個名字，才能在 loop 執行到一半時叫停它，這是人類管理學給 AI 的最好禮物。

**天馬行空的延伸：** 🚀 如果把這十條規則反過來用——設計一個「失敗模式偵測 Agent」，專門掃描其他 Agent 的輸出 log，一旦偵測到 Kitchen Sink 行為模式就自動叫停並回報，這個品管 Agent 本身就是一個可出售的 Skill。

**💬 最有價值的回應：**
> "The document's real contribution is that it gives loops a vocabulary for self-interruption. Without named failure modes, the loop keeps running. With them, it can stop before compounding the mistake."
— @bcherny（Claude Code 創始人，指出「命名」是自動化 loop 的核心管理機制）

**推文連結：** [https://www.techtimes.com/articles/319214/20260628/karpathy-claudemd-grows-ten-rules-new-self-check-protocol-ai-coding-loops.htm](https://www.techtimes.com/articles/319214/20260628/karpathy-claudemd-grows-ten-rules-new-self-check-protocol-ai-coding-loops.htm)

### 2. **levelsio VPS 雲端開發法：Claude Code 開在伺服器上，手機也能繼續寫程式**

**帳號：** @levelsio

**類型：** 💡 工作流優化

**核心方法/技巧：**
- 花 $5/月 在 Hetzner 或 DigitalOcean 開一台 VPS（虛擬私有伺服器）
- 用 SSH（推薦 Termius App）連進伺服器，安裝 Claude Code 或 Cursor CLI
- 讓 Claude 在伺服器上直接寫 code 並啟動 Nginx web server，打開 IP 就看到結果
- 優點：筆電不需要一直開著、無電池焦慮、可從手機 / iPad / 任何裝置接續工作、Claude 能在你睡覺時繼續跑

**為什麼有用：**
這是真正的「AI 永遠在工作」工作流——你不需要在電腦前，Claude Code 在雲端持續運行。levelsio 用這套方法一年下來，幾小時內就在伺服器上完成了整個 3D 互動專案，無需任何部署步驟，因為 Claude 直接在 production 上工作。

**編輯觀點：** 💬 這個方法的最大心理改變是：Claude Code 從「你在用的工具」變成「幫你工作的員工」——你不需要一直守在電腦前，這才是 AI 的正確使用姿勢。

**天馬行空的延伸：** 🚀 多開幾台 $5 VPS，每台跑不同的 Claude Code 專案，像管理一個「AI 外包工作室」——早上出任務，晚上收成果，成本一個月不超過一杯咖啡。

**💬 最有價值的回應：**

— @levelsio（本人分享，描述在伺服器上實時開發的速度感受）

### 3. **Loop Engineering 完整解析：/goal 指令如何讓 AI 知道何時該停下來**

**帳號：** @bcherny（Claude Code 創始人）+ @addyosmani（Google）

**類型：** 🛠️ 可複用方法（「延伸」：Loop Engineering 六月份最完整拆解）

**核心方法/技巧：**
- 核心洞察：LLM 是無狀態的，每次對話都忘記一切，所有持久狀態必須存在外部（CLAUDE.md、git、檔案）
- `/goal` 指令使用獨立的快速驗證模型（非生成 code 的同一個模型）來檢查是否達成目標條件，這是 loop 不失控的關鍵
- 啟動 loop 前的四個決定：①定義可機器驗證的完成條件 ②設定 `--max-turns` 上限 ③分離「生產者」和「檢查者」sub-agent ④用 `--worktree` 隔離並行 Agent
- Token 成本警告：agentic loop 每次工具調用都重新發送完整對話歷史，20 次迭代後每步可超過 50,000 tokens，一個 200 次迭代的任務可能花費 $80+

**為什麼有用：**
「設計 loop」而不是「寫 prompt」是 2026 年 AI 工程師的核心技能轉變。但 loop 不設上限就是一台用高速燒錢的機器——這篇文章提供了最完整的生產級安全 loop 設計指南。

**編輯觀點：** 💬 企業層面最震撼的數據：Uber 四個月燒完全年 AI 預算、Microsoft M365 部門六月提前結束多數 Claude Code 授權——token 計費的衝擊比大多數人預期的快得多。

**天馬行空的延伸：** 🚀 開一個「Loop 計費計算器」小工具，輸入任務類型和預期迭代次數，自動估算 token 費用，並建議最佳的 `--max-turns` 上限——這個工具本身就能靠 Claude Code 在半天內完成。

**💬 最有價值的回應：**
> "I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops."
— @bcherny（Claude Code 創始人，定義 2026 年 AI 工程師的工作本質）

**推文連結：** [https://www.techtimes.com/articles/318828/20260622/claude-code-loop-engineering-stop-prompting-start-designing-autonomous-agent-workflows.htm](https://www.techtimes.com/articles/318828/20260622/claude-code-loop-engineering-stop-prompting-start-designing-autonomous-agent-workflows.htm)

### 4. **emollick 真實 AI Agent 實驗：AI 自主複現學術論文，失敗點不在幻覺**

**帳號：** @emollick（Wharton 商學院教授）

**類型：** 📝 提示技巧 + 思維轉變

**核心方法/技巧：**
- 實驗：給 Claude Sonnet 4.5 一篇複雜經濟學論文全文（含複現資料），不給任何額外指示
- Agent 自主閱讀論文、打開資料壓縮包、分類檔案、將 STATA 統計代碼轉換為 Python、逐步驗證所有研究結論，並成功複現
- 專家 vs AI 對比：AI 在 14 年資深專家設計的複雜任務中「險勝」輸給人類——主要失敗原因是格式問題和未遵循指示，而非幻覺
- 關鍵洞察：「/goal 和全自動 AI Agent 很酷，但不適合所有情境」——何時需要人在 loop 中，是設計問題，不是技術問題

**為什麼有用：**
這個實驗翻轉了很多人對 AI Agent 失敗原因的假設——不是因為「AI 不夠聰明」，而是因為「指示不夠清晰」。這意味著你不需要等更強的模型，現在就可以用 Claude 完成複雜的自主工作，只要你給出可機器執行的指示。

**編輯觀點：** 💬 「格式問題和未遵循指示」是 AI 輸給人類專家的主因——這說明提示詞工程的核心已從「讓 AI 懂你的意思」轉變為「讓 AI 的輸出格式完全符合下一個步驟的輸入格式」，工作流設計比對話技巧更重要。

**天馬行空的延伸：** 🚀 用 Claude 複現本公司過去的成功案例分析報告——把舊報告當作論文、把 CRM 數據當作複現資料，讓 Agent 自動驗證「過去的判斷是否還適用於今天的市場」。

**💬 最有價值的回應：**
> "The margin varies dramatically by industry. In some domains AI won clearly; in others humans won clearly. The question is no longer 'can AI do this' but 'which specific subtasks should AI own.'"
— 研究評審委員會成員（指出 AI 任務擅長領域分化明顯，促使重新思考人機分工）

**推文連結：** [https://www.oneusefulthing.org/p/real-ai-agents-and-real-work](https://www.oneusefulthing.org/p/real-ai-agents-and-real-work)

### 5. **Cloudflare PACT 協議：Chrome、Firefox、Edge 聯手，密碼學 Token 取代 CAPTCHA**

**帳號：** @Cloudflare（機構）

**類型：** 🚀 新工具 + 思維轉變

**核心方法/技巧：**
- 2026 年 6 月 22 日：Cloudflare 聯合 Chrome、Firefox、Edge 宣布開發 PACT（Private Access Control Tokens）協議
- 機制：當瀏覽器偵測到真人使用，自動附加一個匿名 Token；其他網站可驗證此 Token 而無需另做 CAPTCHA
- 隱私保護：Token 無法被用於追蹤用戶或重建瀏覽歷史
- 背景：Cloudflare Radar 數據顯示，當前機器人流量（58%）已超過人類流量（42%）
- 由 Shopify 共同開發，計劃提交為 W3C 標準

**為什麼有用：**
CAPTCHA 已成為 AI 自動化工作流的最大障礙之一——每次 Claude Code 或其他 Agent 嘗試操作網頁時都卡在「我不是機器人」。PACT 協議通過後，合法的 AI Agent（有 Token 授權的）可以無縫通過，自動化工作流的摩擦將大幅降低。

**編輯觀點：** 💬 PACT 協議的意義不只是「告別 CAPTCHA」——它建立了一個「授權 AI Agent」vs「惡意機器人」的區分框架，這是 AI 能被合法整合進 web ecosystem 的重要基礎設施。

**天馬行空的延伸：** 🚀 PACT 普及後，「你的 AI Agent 有沒有瀏覽器授權 Token」將成為新的訂閱分層依據——未來高階 Claude/ChatGPT 訂戶的 AI Agent 可以無障礙瀏覽全網，免費版本的 Agent 仍然被 CAPTCHA 攔截。

**💬 最有價值的回應：**
> "This doesn't just fix CAPTCHAs. It gives the internet a permission layer for AI agents — something we've never had before and desperately needed."
— Shopify 工程師（指出 PACT 解決的是整個 AI Agent 生態系統的身份驗證問題）

**推文連結：** [https://www.techtimes.com/articles/318891/20260623/cloudflare-chrome-firefox-plan-replace-captchas-cryptographic-tokens.htm](https://www.techtimes.com/articles/318891/20260623/cloudflare-chrome-firefox-plan-replace-captchas-cryptographic-tokens.htm)

### 6. **Ornith-1.0：開源自我學習編程模型，會在訓練中改良自己的訓練框架**

**帳號：** @DeepReinforce（開源研究團隊）

**類型：** 🚀 新工具

**核心方法/技巧：**
- 2026 年 6 月 25 日發布：MIT 授權，Hugging Face 可直接下載
- 創新點：模型在強化學習訓練期間，先提議一個任務專屬的「訓練支架（scaffold）」，再在該支架條件下生成解決方案——讓任務特定的編程策略自動湧現
- 四個版本：9B Dense（輕量）、31B Dense、35B MoE、397B MoE（旗艦）
- 旗艦版 397B 在 Terminal-Bench 2.1 得分 77.5、SWE-Bench Verified 得分 82.4，超越 Claude Opus 4.7
- 底層基於 Gemma 4 和 Qwen 3.5

**為什麼有用：**
對於想在本地跑高品質 AI 編程 Agent 的人，Ornith-1.0 是目前最強的開源選項，且 MIT 授權意味著可以商業使用、自由部署。9B 版本在一般消費級 GPU 上可跑，適合個人使用。

**編輯觀點：** 💬 「模型學會改良自己的訓練框架」聽起來很科幻，但實際意義是：這個模型比一般模型更擅長「想清楚解題步驟再動手」，這正是 Karpathy 十條規則第一條想讓 Claude 做到的事——這次是自動學會的。

**天馬行空的延伸：** 🚀 把 Ornith-1.0 的自我scaffolding 機制應用到寫作任務——訓練前先讓 AI 設計「這篇文章的大綱模板」，再在模板框架下生成內容，而不是直接叫 AI 寫文章。這個「兩步驟：先設計框架，再填充內容」的方法論可能對非技術工作者同樣有效。

**💬 最有價值的回應：**
> "The difference from normal RL: instead of learning what answer to give, it also learns how to think about the problem. That's a qualitatively different kind of improvement."
— Hugging Face 社群評論（解釋 self-scaffolding 與傳統強化學習的根本差別）

**推文連結：** [https://www.techtimes.com/articles/319122/20260626/open-source-coding-model-ornith-10-writes-its-own-training-scaffold-reinforcement-learning.htm](https://www.techtimes.com/articles/319122/20260626/open-source-coding-model-ornith-10-writes-its-own-training-scaffold-reinforcement-learning.htm)

### 7. **Perplexity Computer for Counsel：20 個模型同時工作的法律 AI 平台**

**帳號：** @Perplexity

**類型：** 🚀 新工具（多模型路由架構示範）

**天馬行空的延伸：** 🚀 用類似概念設計「內容創作 Computer for Counsel」：研究子任務用 Perplexity、事實核查子任務用 Claude、文案潤稿用 GPT-5.6、SEO 關鍵詞用 Gemini，並讓每一步輸出都附上來源鏈接——這樣你的文章就有 AI 審計軌跡。

**💬 最有價值的回應：**
> "20 different models integrated — when you release these sub-agents, it's like releasing a team. The best agent that does the best thing will be assigned a certain task."

### 12. **HTTP QUERY 方法正式引入：帶數據體的 GET，解決大量參數傳遞問題**

**帳號：** @ruanyf（阮一峰周刊工具推薦）

**類型：** 🛠️ 技術工具

**核心方法/技巧：**
- HTTP 正式在 GET 和 POST 之外引入 QUERY 方法
- 核心特性：可以攜帶請求數據體（Request Body），但不會被服務器緩存
- 解決的問題：需要傳遞大量過濾參數時，GET 的 URL 長度限制 + POST 的語義問題——QUERY 兼顧兩者優點
- 對 AI Agent 的意義：AI 在調用外部 API 時，可以用 QUERY 方法傳遞複雜的結構化查詢參數

**為什麼有用：**
這是 Web 標準的重要補完，特別是對於需要大量查詢參數的 AI Agent API 調用場景（如語義搜索、複雜篩選）。如果你在設計 AI 工作流的 API 接口，QUERY 方法值得納入考量。

**編輯觀點：** 💬 一個新 HTTP 方法的誕生通常意味著某個長期存在的 workaround 即將退出歷史舞台——「把 GET 參數塞進 POST Body」的反模式終於有官方解法了。

**天馬行空的延伸：** 🚀 為 Claude Code 或任何 AI Agent 設計 API 時，把語義搜索查詢參數（embedding vectors、過濾條件、排序規則）放進 QUERY 方法的 Body，讓 URL 保持乾淨，同時確保複雜查詢不會被錯誤緩存。

**💬 最有價值的回應：**
> "QUERY is basically a GET with a body — semantically correct, not cached, not confused with mutation. We've needed this since REST became a thing."
— Web 標準社群評論（解釋 QUERY 方法為何姍姍來遲但恰逢其時）

**推文連結：** [https://kreya.app/blog/new-http-query-method-explained/](https://kreya.app/blog/new-http-query-method-explained/)

### 13. **Cloudflare + Mozilla PACT 協議技術細節：匿名令牌如何在不追蹤的前提下驗證身份**

**帳號：** @Cloudflare

**類型：** 📝 技術理解

**核心方法/技巧：**
- Cloudflare 和 Mozilla 解釋 PACT 的技術機制：當瀏覽器確認是真人操作，生成一個密碼學匿名 Token
- Token 設計：不含用戶識別信息，無法被用於跨站追蹤，基於零知識證明原理
- 對 AI Agent 生態的意義：計劃支援「授權 AI Agent」獲得 Token，將合法的自動化工具與惡意爬蟲區分開來
- Mozilla 提供完整技術說明文件：[hacks.mozilla.org/2026/06/pact-anonymous-credentials-for-the-web/](https://hacks.mozilla.org/2026/06/pact-anonymous-credentials-for-the-web/)

**為什麼有用：**
理解 PACT 機制可以幫助你預判未來的網路訪問規則——擁有授權 Token 的 AI Agent 可以無障礙訪問網站，沒有 Token 的會繼續被 CAPTCHA 阻擋。對於設計需要大量網頁訪問的 AI 工作流，現在就應該規劃如何在 PACT 生態中獲得授權。

**編輯觀點：** 💬 零知識證明加上密碼學 Token 這種組合，讓「我是真人/合法 Agent」的驗證不需要交出任何個人信息——這和 Anthropic 要求人臉掃描的策略形成了有趣的對比，同樣是身份驗證，哲學截然不同。

**天馬行空的延伸：** 🚀 未來可能出現「AI Agent 身份登記所」——組織在其中登記自己的 Agent、取得 PACT Token、並對 Agent 行為負責，就像公司登記制度一樣。你的個人品牌 AI Agent 會有一個「商業登記號碼」。

**💬 最有價值的回應：**
> "Bots now outnumber humans online 58% to 42%. PACT isn't a nice-to-have — it's an infrastructure requirement for keeping the web functional."
— Cloudflare Radar 研究員（提供讓人警醒的機器人流量比例數據）

**推文連結：** [https://hacks.mozilla.org/2026/06/pact-anonymous-credentials-for-the-web/](https://hacks.mozilla.org/2026/06/pact-anonymous-credentials-for-the-web/)

### 15. **SlopGuard：AI 自動給 GitHub PR 打品質分，過濾 AI 生成的低質量提交**

**帳號：** @ruanyf（阮一峰周刊工具推薦，作者 @Blue-B）

**類型：** 🚀 新工具

**核心方法/技巧：**
- SlopGuard 是一個 GitHub App，安裝後自動對每個 PR 和 Issue 打品質分
- 主要解決：AI Agent 大量生成代碼的時代，低質量提交（「Slop」）佔用 Code Review 時間
- 品質評分維度：代碼可維護性、測試覆蓋、文檔完整性、提交信息品質
- 評分結果標注在 PR 上，讓維護者可以快速過濾，優先處理高品質貢獻
- MIT 授權，GitHub：[github.com/Blue-B/slopguard](https://github.com/Blue-B/slopguard)

**為什麼有用：**
當 Claude Code、Codex 等工具讓每個人都能快速生成並提交 PR 時，Code Review 的瓶頸從「寫程式」變成了「判斷哪些程式值得 Review」。SlopGuard 是應對這個新問題的直接工具。

**編輯觀點：** 💬 「Slop」這個詞（意指 AI 生成的無品質感大量輸出）在 2026 年已成為技術社群的日常詞彙，有人專門做工具對抗它——這本身就說明 AI 生成垃圾的速度比人類消化的速度快太多了。

**天馬行空的延伸：** 🚀 把 SlopGuard 的評分機制應用到內容創作：設計一個「Content Guard」，每次 AI 幫你生成文章草稿時自動評估「原創性分數」、「觀點深度分數」、「可執行性分數」——低於門檻的草稿直接要求 AI 重做，不進入人工編輯環節。

**💬 最有價值的回應：**
> "We went from 'write more code' being the bottleneck to 'review more code.' SlopGuard is the triage layer we needed before the queue was overflowed."
— 開源維護者評論（描述 AI coding 時代 Code Review 瓶頸的轉移）

**推文連結：** [https://github.com/Blue-B/slopguard](https://github.com/Blue-B/slopguard)
