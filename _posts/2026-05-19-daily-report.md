### 1. **多步驟 Agent 工作流的三個必備設計原則**

**帳號：** @swyx（Shawn Wang）

**類型：** 🛠️ 可複用方法

**核心方法/技巧：**

- **Checkpoint 機制**：每個子任務完成後輸出結構化 JSON 狀態，讓 Agent 在出錯時可從斷點重啟，而非全程重跑
- **Tool 分層設計**：將工具分為「讀取類」（無副作用）與「寫入類」（有副作用），讀取類自動執行，寫入類必須人工確認
- **Fallback 降級策略**：主力模型失敗時自動切換至輕量模型完成剩餘步驟，確保工作流不中斷

**為什麼有用：**

大多數 Agent 工作流在生產環境中失敗，不是因為模型不夠強，而是缺乏容錯設計。這三個原則可以直接套用到任何現有 n8n、Dify 或自建 Agent 流程中。

**編輯觀點：** 💬 Checkpoint 機制是最容易被忽略但回報最高的設計——一個 50 步工作流只要在第 48 步出錯就得全部重來，這個痛點幾乎所有人都踩過。

**天馬行空的延伸：** 🚀 把 Checkpoint 機制應用到創作流程本身——每完成一個段落就輸出一個「創作狀態快照」，讓 AI 助理能在你明天繼續時無縫銜接昨天的思路脈絡。

**💬 最有價值的回應：**

> 補充一點：在「讀取類」vs「寫入類」工具的分類上，我建議再加一個「不可逆寫入類」（如發送郵件、發布內容），這類工具除了人工確認外，還應加上 dry-run 預覽模式，讓用戶先看到執行結果再決定是否真的送出。

### 2. **內容創作者的 Prompt 三層框架：角色 × 情境 × 約束**

**帳號：** @danshipper（Dan Shipper）

**類型：** 📝 提示技巧

**核心方法/技巧：**

- **第一層：角色（Role）**：不只說「你是一個作家」，而是給出具體的聲音特徵——「你是一個擅長用具體細節代替抽象概念的非虛構作家，文風接近 Paul Graham 但更口語化」
- **第二層：情境（Context）**：提供讀者畫像、發布平台、內容目標，讓 AI 理解「為誰寫、在哪寫、為何寫」
- **第三層：約束（Constraint）**：明確列出禁止事項比允許事項更有效——「不要使用任何含有『賦能』『生態系統』『顛覆』的詞彙」

**為什麼有用：**

大多數 Prompt 只用了第一層（角色），導致輸出內容雖然方向對但細節模糊。加入情境與約束層後，內容的品牌一致性和可用率能顯著提升。

**編輯觀點：** 💬 「約束層」的反向設計思維是關鍵洞見——告訴 AI 不要做什麼，往往比告訴它要做什麼更精準，因為禁止清單可以直接反映你踩過的坑。

**天馬行空的延伸：** 🚀 把這個三層框架做成你的個人「品牌 Prompt 模板庫」，每次為不同平台（LinkedIn、Newsletter、Podcast 講稿）切換時只需替換情境層，角色層和約束層保持不變，建立跨平台的聲音一致性。

**💬 最有價值的回應：**

> 我在實際使用中發現，「約束層」最好從自己過去最不滿意的 AI 輸出中提煉——把那些讓你翻白眼的表達全部列進去。這樣做出來的約束清單是最個人化、最有效的。

— @emollick（分享了親身案例，提出「從失敗案例反推約束清單」的具體操作路徑，極具實踐價值）

### 3. **用 Claude Projects 打造個人知識庫助理的完整流程**

**類型：** 💡 工作流優化

- **結構化上傳**：將筆記、文章草稿、過去作品分資料夾上傳，並在每份文件頂部加入 `# 文件類型：[筆記/草稿/已發布]` 標記，讓 AI 能區分「參考用」與「模仿用」素材

**為什麼有用：**

Claude Projects 的知識庫功能被大多數人當成「搜尋引擎」使用，但它真正的價值在於「思維夥伴」——能跨越你所有的碎片化輸入，找出你自己都沒意識到的思考模式。

**編輯觀點：** 💬 「跨文件連結發現」這個用法讓我眼睛一亮——這不只是整理工具，而是把 AI 當成外腦的第二雙眼，幫你看見自己思維中的盲點和潛在主題。

**天馬行空的延伸：** 🚀 把這個工作流擴展為「內容預測系統」——定期詢問 AI「根據我的思考模式，我接下來最可能想寫什麼？」讓過去的創作反哺未來的選題決策。

### 4. **本地 LLM + 自動化腳本：零成本的每日簡報系統**

**類型：** 🚀 新工具

**核心方法/技巧：**

- **工具組合**：Ollama（本地跑 Llama 3 或 Mistral）+ Python 腳本 + RSS 訂閱源，每天自動抓取指定來源並生成個人化摘要
- **成本控制**：完全本地執行，無 API 費用；M2 MacBook 跑 7B 模型每次摘要約 15 秒，完全可接受
- **個人化過濾器**：腳本中加入關鍵字黑名單（如「融資」「估值」「IPO」），自動過濾掉不關心的商業新聞，只保留技術和產品相關內容

**編輯觀點：** 💬 最妙的是「黑名單過濾器」的設計思想——與其定義你想看什麼，不如定義你不想看什麼。這個逆向思維在信息過載時代尤為珍貴。

### 5. **AI 產品的 48 小時快速驗證法：從想法到用戶反饋**

**帳號：** @karpathy（Andrej Karpathy）

**類型：** 🛠️ 可複用方法

**核心方法/技巧：**

- **Hour 0-8：單頁驗證**：用 Claude 或 GPT-4o 生成一個靜態 HTML 頁面，模擬核心功能的 UI，不寫任何後端代碼；直接分享給 10 個目標用戶請他們「假裝」使用
- **Hour 8-24：痛點萃取**：收集用戶在「假裝使用」時口頭表達的困惑和期待，提取出 3 個最高頻的需求點

**為什麼有用：**

傳統 MVP 方法論要求你先建構再驗證，但 AI 時代讓「驗證先於建構」成為可能。用靜態 HTML 模擬功能幾乎不需要技術成本，卻能獲得 80% 的驗證價值。

**編輯觀點：** 💬 「假裝使用」這個設計讓人眼前一亮——不需要功能真的能用，只需要界面足夠真實，就能誘發用戶的真實心理反應。這把用戶研究的門檻降到了幾乎為零。

**天馬行空的延伸：** 🚀 把「靜態 HTML 模擬」的思路用在內容創作上：在動筆前先用 AI 生成一個「假文章大綱+假金句」的模擬版，分享給受眾預覽，根據他們的反應決定是否值得花時間深入寫作。

> 這個方法我在上個月一個 SaaS 項目上親測有效。補充一個細節：「假裝使用」環節中，讓用戶邊操作邊說出心裡在想什麼（think-aloud protocol），比事後問問卷收集到的信息質量高出數倍。
